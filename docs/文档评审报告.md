## 二、严重问题（M1 前必须解决）

### 问题 1：全互联拓扑下事件传播存在设计缺陷（§5.5）

当前设计用 source_id + content_hash + 冷却窗口防回环。但在 3 台以上设备全互联时：

- A 复制 → 发给 B 和 C
- B 写入剪贴板 → B 的监听器触发 → B 发给 C
- C 收到两份相同内容（来自 A 和 B）

content_hash 去重窗口可缓解，但会引入假阳性（用户在窗口内合法复制相同内容被丢弃）。

**建议**：采用基于 event_id 的"已见集合"（布隆过滤器或 LRU 集合），收到事件先检查 event_id 是否已见。

### 问题 2：TCP 连接模型不明确（§4.2、§5.3）

文档说"每节点监听 + 主动连接"，暗示每对设备可能两条 TCP 连接。10 台设备若每对两条 = 90 条连接，资源浪费且状态管理复杂。

**建议**：明确"每对一条连接"，使用 device_id 比较做 tie-breaking（较小者做服务端）。

### 问题 3：认证 salt 策略存在安全缺陷（§5.4）

文档说 salt "每设备或每部署随机"或"固定应用 salt"——两者矛盾：

- 每设备随机 salt → 其他设备无法派生相同 root_key → 认证无法完成
- 固定应用 salt → Argon2id 失去抗彩虹表能力

**建议**：评估 SPAKE2 或 CPace 等 PAKE 协议，或明确 salt 通过带外方式在所有设备间共享。

## 三、中等问题（M2 前解决）

### 问题 4：缺少 CI/CD 与发布管道设计

文档在 M0 中提到 CI（fmt/clippy/test），但没有具体说明：

- 构建矩阵（Windows + Linux，x86_64）
- 跨平台编译策略
- 二进制分发方式（GitHub Releases？）
- Windows 安装器 vs 便携版
- Linux 打包格式（.deb？AppImage？静态二进制？）
- 代码签名（Windows SmartScreen 会阻止未签名二进制）

### 问题 5：缺少剪贴板格式跨平台映射

第 6 章定义了 ClipboardBackend trait，但没有讨论跨平台同步时的格式映射：

- Windows 使用 CF_UNICODETEXT、CF_HDROP、CF_DIB 等格式
- Linux 使用 text/plain、text/uri-list、image/png 等 MIME 类型
- 从 Windows 同步文件列表到 Linux 时，CF_HDROP 如何转换为 text/uri-list？
- 从 Linux 同步图片到 Windows 时，如何选择 CF_DIB vs CF_BITMAP？

### 问题 6：防火墙/NAT 处理缺失

即使在局域网内，Windows Defender 防火墙默认会阻止未注册的 TCP 监听端口。应用程序需要在首次运行时注册防火墙规则或引导用户操作。Linux 上 iptables/nftables 同样可能阻止端口。文档完全没有涉及这个部署层面的关键问题。

### 问题 7：里程碑时间估算不足

仅 M0 标注了"约 1 周"，M1-M4 没有时间估算。根据模块复杂度分析：

| 里程碑 | 内容 | 预估时间（单人全职） |
|--------|------|---------------------|
| M1 | 文本同步 MVP（TCP 协议、加密握手、双平台剪贴板、防回环、重连） | 3–4 周 |
| M2 | UDP 发现 + 设置 UI | 1–2 周 |
| M3 | 图片/文件 + 历史 | 3–4 周 |
| M4 | 打磨发布 | 2–3 周 |
| **总计** | | **约 10–14 周** |

M1 的工作量被严重低估，实际远不止文档暗示的范围。

## 四、轻微问题（M3+ 解决）

### 问题 8：egui-tokio 桥接模式未具体化

文档提到"UI 线程与异步桥接需明确边界"但未给出具体方案。egui 是即时模式、单线程的，与 tokio 的异步世界桥接是已知的复杂点。建议在 M0 阶段就确定具体模式（如 `Arc<Mutex<Vec<Event>>>` + `ctx.request_repaint()` 轮询，或使用 `eframe::Frame` 的 `send_event` 机制）。

### 问题 9：缺少升级/迁移策略

配置文件格式（TOML）和数据格式（SQLite 或自定义）在未来版本可能变化。文档没有讨论向后兼容性策略或数据迁移机制。

### 问题 10：README 过于简略

当前 README 仅 23 行，缺少构建说明、快速入门、架构图、UI 原型截图等。对于"规划中"阶段可接受，但应在 M0 完成后大幅扩充。

## 七、补充风险

除文档已列出的 7 项风险外，建议增加：

| 新增风险 | 影响 | 对策 |
|----------|------|------|
| 全互联事件重复传递 | 剪贴板重复写入、带宽浪费 | event_id 已见集合 + 布隆过滤器 |
| TCP 连接风暴（网络恢复时） | 瞬时连接失败、CPU 飙升 | 重连时添加随机抖动（jitter） |
| 密码变更后节点锁定 | 已配对设备全部失联 | 密码变更时广播通知；提供"重置配对"功能 |
| 剪贴板写入竞态 | 多设备同时写入导致闪烁或丢失 | 写入队列 + 最后写入者胜出策略 |
| tokio 运行时关闭挂起 | 进程无法正常退出 | 设置 shutdown 超时 + 强制退出 |
| 历史记录内存累积 | 图片历史导致 RAM 超限 | 内存中仅保留元数据和缩略图，原图按需加载 |
| egui N100 渲染性能 | 大量历史条目时帧率下降 | 虚拟列表 + 分页加载 |
| Windows SmartScreen 拦截 | 用户无法运行未签名二进制 | 代码签名或提供安装器绕过 |

## 八、总结

OhMyCopy 的开发文档是一份**质量中上**的规划文档。在项目处于"零代码、纯规划"阶段的前提下，展现了成熟的工程思维。文档的突出优势在于：范围控制严格、KPI 可量化、协议设计专业、技术选型合理、决策记录完善。

**最需要优先解决的三个核心问题**：

1. **全互联拓扑下的事件传播策略**（§5.5）—— 当前基于 source_id + content_hash 的方案无法正确处理 3 个以上节点的场景
2. **TCP 连接模型**（§4.2、§5.3）—— 需明确"每对一条连接"并使用 tie-breaking 规则
3. **认证方案的 salt 处理**（§5.4）—— salt 策略前后矛盾，影响认证正确性

这三个问题直接影响 M1 核心同步功能的正确性，应在编码前彻底澄清。此外，里程碑的时间估算需要重新校准——M1 涵盖了协议实现、加密、双平台剪贴板适配和可靠性机制，实际工作量远超文档暗示的范围。

总体而言，这份文档为项目提供了一个**坚实的基础**，经过上述改进后，可以成为一份优秀的技术规格说明。
